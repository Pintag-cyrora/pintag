# Department Graduation Checklist

*Every department uses this same checklist before advancing to the next Department Maturity level (see `DEPARTMENTS.md`). Copy the checkboxes below into the department's own Playbook (§20, Department Maturity) and check them off honestly, with evidence — not because software exists, but because the department is being successfully operated.*

**Department maturity is measured by operation, not implementation.** Writing software is only one step. Operating the department successfully — daily, by us, before anyone else — is what proves the software is correct.

---

- [ ] **Playbook approved.** The department's Playbook (`departments/<name>/PLAYBOOK.md`) is complete against `_TEMPLATE.md`'s structure and the founder has signed off on it.
- [ ] **Required software implemented.** Every item in the Playbook's §16 Required Software backlog that's marked as needed for this maturity level actually exists and works.
- [ ] **End-to-end workflow operational.** The department's core SOP (§17) has been run start to finish, for real, and produced a correct result — not just unit-tested in isolation.
- [ ] **Used daily for at least two consecutive weeks.** Real elapsed time, real usage, not a single successful dry run. This is the one criterion that can't be shortcut by writing more code faster.
- [ ] **At least one improvement made based on real-world usage.** Something about the workflow or the software changed because operating it revealed a gap a design session didn't — the whole point of using it ourselves before offering it externally.
- [ ] **KPIs actively tracked.** The Playbook's §11 KPIs are being measured against real numbers, on a real cadence, not left as an example list.
- [ ] **Founder sign-off.** Explicit confirmation that this department is ready to be relied on daily going forward.

A department does not graduate because software exists. It graduates because it is being successfully operated.
