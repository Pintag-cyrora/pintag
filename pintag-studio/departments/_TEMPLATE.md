# [Department Name] — Playbook

*A Department Playbook is not documentation. It is the specification software gets built from.*

*Old workflow: Idea → Software. New workflow (Department-Driven Development — see `../DEPARTMENTS.md`): Department → Playbook → Workflow → Software → Daily Operation → Improvement → Standard → Next Department. Software is always an implementation of this Playbook — never the other way around. Nothing gets built for this department until the relevant section below is written and true.*

*Copy this file to `departments/<name>/PLAYBOOK.md` and fill in every section. Leave a section explicitly marked "Not yet defined" rather than deleting it or faking an answer — an honest gap is more useful than a confident guess. See `departments/intelligence/PLAYBOOK.md` for a filled-in example.*

---

## 1. Mission

Why does this department exist? What business problem does it solve? Why is it important?

## 2. Customer

Every department serves another department. Who does this one exist to help — which department (or departments) receives its output? A department that can't name its customer doesn't yet know why it exists.

## 3. Vision

What does world-class look like for this department? Describe the ideal future state.

## 4. Responsibilities

Everything this department owns. Clearly define boundaries — what is *not* this department's responsibility, and whose is it instead?

## 5. Inputs

What information does this department receive? (e.g. business goals, research, knowledge, brand, customer feedback, performance data.)

## 6. Outputs

Everything this department produces. (e.g. reports, campaigns, content, recommendations, strategies, knowledge.)

## 7. Daily Routine

What should this department do every morning, afternoon, evening? Think of this as the AI employee's daily operating procedure. If nothing runs daily yet, say so — don't describe an aspiration as if it's already running.

## 8. Weekly Routine

Weekly planning, reviews, reports, meetings, optimization.

## 9. Monthly Routine

Campaign planning, performance review, strategy review, knowledge review, brand review.

## 10. Quarterly Routine

Goals, major initiatives, roadmap review, market review, business review.

## 11. KPIs

How do we measure success? Every department needs measurable outcomes — name the actual numbers, not just the categories.

## 12. Decision Authority

What decisions can this department make independently? What requires another department? What requires human (founder) approval?

## 13. Collaboration

Which departments does this one work with? What are the handoffs, in and out?

## 14. AI Agents

Which AI agents belong to this department? Current agents and their responsibilities. Future agents, if planned. If none exist yet, say so.

## 15. Tools

*External* tools and services this department uses or depends on — distinct from AI Agents (who does the work) and Required Software (what Marketing OS itself must build). e.g. Google Trends, Meta Ads Library, Canva, CapCut, GA4, a CRM. List only what's actually in use today; name planned tools separately as "not yet adopted."

## 16. Required Software

What software must exist to support this department? Dashboards, views, automation, reports, approval flows, integrations. **This section is the department's development backlog** — nothing here gets built until it's written here first.

## 17. Standard Operating Procedures

Document the department's major workflows as explicit step sequences (e.g. `Detect trend → Score relevance → Recommend campaign → Create brief → Send to Strategy`). These SOPs should eventually become automations — write them precisely enough that they could be.

## 18. Intelligence Contribution

How does this department improve Marketing OS's Intelligence Layer? (e.g. creates knowledge, updates brand memory, learns from performance, improves language, finds trends, corrects marketing patterns.) Every department should contribute back — if this one doesn't yet, say what it should contribute once it can.

## 19. Definition of Done

When can we consider this department operational? List clear, checkable completion criteria — not a feeling, a checklist.

## 20. Department Maturity

Rate this department honestly against the standard model (see `../DEPARTMENTS.md`):

- **Level 0 — Not Defined**
- **Level 1 — Playbook Complete**
- **Level 2 — Software Exists**
- **Level 3 — Operated Daily**
- **Level 4 — Standardized**
- **Level 5 — Self-Improving**

**Current level:** [state it, with one sentence of evidence]
